#45 - Mobile navigation bar redesign

What & Why

The current bottom tab bar has 6 tabs which is cluttered on small screens. Redesign to show exactly 5 items: Prayer, Qur'an, Qibla (centre — promoted as the focal point with a larger decorative icon), Events, and a Burger/More button. The burger opens a slide-in drawer with Islamic decoration that houses the remaining items (Blog, Alerts/notifications settings, and any future items) — keeping the primary bar clean and the secondary navigation beautiful.

Done looks like

Bottom bar shows exactly: Prayer | Qur'an | Qibla (elevated centre button) | Events | ☰
The Qibla button is visually distinct — larger, gold accent, slightly raised above the bar
Tapping ☰ slides in a drawer from the right (or bottom sheet) with:
A decorative Islamic geometric header in brand-green with gold accents
Blog entry
Alerts (notification settings) entry
Each row has an icon and label, tapping navigates and closes the drawer
Both iOS NativeTabs path and Android/web ClassicTabs path are updated
No existing screens are removed — they all remain navigable
Out of scope

Redesigning the individual screens
Adding new screens or features
Web layout changes
Steps

Build an AppDrawer component — a Modal or Animated slide-in panel (bottom sheet style or right-slide). Render a decorative header: deep-green background, gold geometric Islamic star/pattern overlay, "Menu" title. Below it render rows for Blog and Alerts (each with icon + label). Tapping any row navigates via router.push(...) and closes the drawer. Use useSafeAreaInsets for safe area padding.
Redesign ClassicTabLayout — reduce to 5 Tabs.Screen entries: index (Prayer), quran, qibla (centre), events, and a dummy "more" screen. Override the More tab icon with a hamburger icon. Intercept its tabBarButton press to open the AppDrawer instead of navigating — using a TouchableOpacity wrapper that sets drawer state. Hide the blog and settings Tabs.Screen entries from the bar (use tabBarItemStyle: { display: 'none' }) while keeping them navigable.
Redesign NativeTabLayout — mirror the same 5-item order. Since NativeTabs doesn't support a fake tab, render a custom tabBar overlay button for the More slot that opens the AppDrawer. If that API is too limited, fall back to using ClassicTabLayout for all platforms so both paths use the same custom bar.
Style the Qibla centre button — in ClassicTabLayout give the qibla tab a custom tabBarButton that renders a raised circular gold-bordered button slightly above the bar baseline. In NativeTabs, use a larger SF Symbol with the accent colour.
Wire and test — confirm Blog and Alerts screens are still reachable via the drawer, deep-links still work, and the onboarding flow is unaffected.

Plan #48 - Admin image uploads for content
#48 - Admin image uploads for content

What & Why

The admin forms for Events, Announcements, and Blog Posts currently use a plain text input for imageUrl / featureImageUrl. Replace this with a proper file-upload widget in every create/edit form so admins can upload an image directly from their device. The uploaded image must be stored via object storage and displayed on the corresponding public-facing page alongside the content.

Object storage (useUpload hook + signed-URL API) is already in place. All three DB columns already exist. This task wires the upload UI into the admin forms and ensures the image renders correctly on the frontend.

Done looks like

Events admin (/admin/events) — create and edit forms show a file upload widget instead of a URL text box. After upload, a thumbnail preview appears in the form. The uploaded image displays on the public events list and event detail page. Helper text reads: "Recommended: 1200 × 630 px (16:9 landscape)"
Announcements admin (/admin/announcements) — same file upload widget. Image renders in the announcement card on the public announcements page. Helper text: "Recommended: 800 × 450 px (16:9)"
Blog post editor (/admin/blog/new and /admin/blog/:id/edit) — feature image field replaced with file upload. Image renders as the full-width hero above the article body. Helper text: "Recommended: 1200 × 630 px (16:9 landscape)"
Accepted formats shown in the UI: JPG, PNG, WebP — max 5 MB
Uploading shows a progress indicator; errors surface inline
If an existing record already has an imageUrl the current image is shown as a preview with an option to replace it
Out of scope

Image cropping or resizing in-browser
Video or audio attachments
Bulk image replacement across existing records
Gallery / multi-image support
Steps

Shared ImageUpload component — Build a reusable <ImageUpload> component in the website's components/ folder. It wraps the existing useUpload hook, accepts an aspectHint prop (shown as helper text), renders a drag-and-drop zone with a file picker fallback, shows an upload progress bar, and displays a thumbnail preview once complete. On success it calls onUploaded(url: string).
Events admin form — Replace the URL text input in the events create/edit form with <ImageUpload aspectHint="Recommended: 1200 × 630 px (16:9 landscape)" />. Wire the returned URL into the form state and submit it as imageUrl.
Announcements admin form — Same substitution with aspectHint="Recommended: 800 × 450 px (16:9)".
Blog post editor — Replace the feature-image URL field with <ImageUpload aspectHint="Recommended: 1200 × 630 px (16:9 landscape)" /> wired to featureImageUrl.
Public Events pages — Ensure the event card and event detail page render the imageUrl when present (check if already wired; add if missing).
Public Announcements page — Ensure announcement cards render imageUrl when present.
Public Blog pages — Ensure the blog post detail page renders featureImageUrl as a full-width hero image above the article body; blog list cards should show a thumbnail if present.
Admin audit pass — While touching each admin section, verify the full CRUD flow (create, read, update, delete, publish toggle) works end-to-end and fix any obvious breakages found during the audit.
